# USERFLOW.md

## 목적
- 유저 도메인의 유저 생성, 타 유저 조회, 본인 조회, 유저 정보 수정, 유저 삭제 흐름을 문서화한다.

## 유저 관리 전략
- 유저는 `email`, `password`, `username`, `deleted` 값을 가진다.
- 이메일과 유저명은 중복될 수 없다.
- 비밀번호는 인코딩 후 저장한다.
- 유저 삭제는 DB row를 제거하지 않고 `deleted` 값을 변경하는 소프트 삭제 방식이다.
- 삭제된 유저는 일반 조회, 수정, 삭제, 인증 이후 서비스 로직에서 active user로 취급하지 않는다.
- 존재하지 않는 유저와 삭제된 유저는 동일하게 `USER_NOT_FOUND`로 처리한다.
- 유저 삭제 시 해당 유저의 Redis RefreshToken 세션을 전체 삭제하여 추가 재발급을 차단한다.
- 인증이 필요한 유저 API는 AccessToken 인증 정보에서 로그인 유저 ID를 추출해 처리한다.
- AccessToken은 Redis에 저장하지 않으므로 유저 삭제 직후에도 만료 전까지 토큰 자체의 서명 검증은 통과할 수 있다.
- 따라서 보호 API에서는 서비스 계층에서 유저 존재 여부와 삭제 여부를 다시 검증한다.

## API 인증 구분
- `POST /user`는 공개 API다.
- `GET /user/{userId}`는 공개 API다.
- `GET /me`는 AccessToken 인증이 필요한 보호 API다.
- `PATCH /me`는 AccessToken 인증이 필요한 보호 API다.
- `DELETE /me`는 AccessToken 인증이 필요한 보호 API다.

## 요청 흐름

### 1. 유저 생성
- 대상 엔드포인트: `POST /user`

동작:
1. 요청값을 검증한다.
2. 이메일 중복 여부를 검사한다.
3. 유저명 중복 여부를 검사한다.
4. 비밀번호를 인코딩한다.
5. `deleted = false` 상태의 유저를 생성한다.
6. 유저를 저장한다.
7. 저장된 유저 정보를 응답한다.

결과:
1. 새로운 유저가 생성된다.
2. 생성된 유저는 로그인 가능한 active user 상태가 된다.
3. 응답에는 비밀번호를 포함하지 않는다.

### 2. 타 유저 조회
- 대상 엔드포인트: `GET /user/{userId}`

동작:
1. 요청 경로의 `userId`로 조회 대상 유저를 식별한다.
2. 조회 대상 유저가 존재하는지 검증한다.
3. 조회 대상 유저가 삭제되지 않았는지 검증한다.
4. 조회 대상 유저 정보를 응답한다.

결과:
1. 삭제되지 않은 유저의 공개 가능한 정보만 응답한다.
2. 이메일과 비밀번호는 응답하지 않는다.
3. 존재하지 않는 유저와 삭제된 유저는 조회할 수 없다.

### 3. 본인 조회
- 대상 엔드포인트: `GET /me`

동작:
1. AccessToken 인증 정보에서 로그인 유저 ID를 추출한다.
2. 로그인 유저가 존재하는지 검증한다.
3. 로그인 유저가 삭제되지 않았는지 검증한다.
4. 본인 유저 정보를 응답한다.

결과:
1. 로그인한 유저는 자신의 상세 정보를 조회할 수 있다.
2. 본인 조회 응답에는 이메일을 포함한다.
3. 비밀번호는 응답하지 않는다.

### 4. 유저 정보 수정
- 대상 엔드포인트: `PATCH /me`

동작:
1. AccessToken 인증 정보에서 로그인 유저 ID를 추출한다.
2. 요청값을 검증한다.
3. 로그인 유저가 존재하는지 검증한다.
4. 로그인 유저가 삭제되지 않았는지 검증한다.
5. 요청한 유저명이 기존 유저명과 같은지 확인한다.
6. 기존 유저명과 다르면 유저명 중복 여부를 검사한다.
7. 유저명을 수정한다.

결과:
1. 로그인 유저의 유저명이 변경된다.
2. 기존 유저명과 동일한 값으로 수정 요청하면 중복 검사를 수행하지 않는다.
3. 이미 다른 유저가 사용하는 유저명으로는 변경할 수 없다.

### 5. 유저 삭제
- 대상 엔드포인트: `DELETE /me`

동작:
1. AccessToken 인증 정보에서 로그인 유저 ID를 추출한다.
2. 로그인 유저가 존재하는지 검증한다.
3. 로그인 유저가 삭제되지 않았는지 검증한다.
4. 유저를 삭제 상태로 변경한다.
5. Redis에 저장된 해당 유저의 RefreshToken 세션을 전체 삭제한다.

결과:
1. 유저는 소프트 삭제 상태가 된다.
2. 삭제된 유저는 active user 검증을 통과할 수 없다.
3. 삭제된 유저의 RefreshToken은 재발급에 사용할 수 없다.
4. 이미 발급된 AccessToken은 만료 전까지 토큰 자체의 서명 검증을 통과할 수 있으므로, 보호 API에서는 유저 삭제 여부를 계속 검증해야 한다.

## 기대 효과
- 유저 삭제 이후에도 기존 데이터는 보존된다.
- 유저 존재 여부와 삭제 여부 검증을 서비스 계층에서 일관되게 처리할 수 있다.
- 본인 조회, 수정, 삭제는 JWT 인증 정보의 userId를 기준으로 처리되므로 요청자가 임의의 userId를 전달하지 않는다.
- 타 유저 조회와 본인 조회의 응답 범위를 분리하여 이메일 노출 범위를 제한한다.
- 유저 삭제 시 RefreshToken 세션을 전체 삭제하여 추가 토큰 재발급을 차단할 수 있다.

## 주의사항
- 유저 삭제는 DB row 삭제가 아니라 소프트 삭제다.
- 이메일과 유저명은 DB unique 제약이 있으므로 삭제된 유저의 이메일과 유저명도 재사용할 수 없다.
- 삭제된 유저를 다시 활성화하는 흐름은 현재 제공하지 않는다.
- AccessToken 인증은 토큰 자체의 유효성만 보장하므로, 보호 API에서 유저 존재 여부와 삭제 여부 검증을 생략하면 안 된다.
- 유저 삭제 API는 RefreshToken 쿠키 만료 응답을 직접 내려주지 않는다. Redis RefreshToken 세션 삭제로 재발급을 차단한다.
- 타 유저 조회는 공개 API이므로 응답 필드에 민감한 정보를 포함하지 않아야 한다.
