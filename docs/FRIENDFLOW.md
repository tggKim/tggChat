# FRIENDFLOW.md

## 목적
- 친구 도메인의 주요 요청 흐름과 정책을 정리한다.

## 친구 관리 전략
- 친구 관계는 단방향 관계다.
- 친구 관계는 `owner`와 `friend`로 표현한다.
- `owner`는 친구를 추가한 유저다.
- `friend`는 친구로 추가된 유저다.
- 유저 A가 유저 B를 친구로 추가해도, 유저 B의 친구 목록에 유저 A가 자동으로 추가되지는 않는다.
- 친구 추가는 로그인 유저 기준으로만 수행한다.
- 친구 추가 대상은 username으로 식별한다.
- 삭제된 유저는 친구로 추가할 수 없다.
- 삭제된 유저는 친구 목록 조회 결과에서 제외된다.
- 같은 owner와 friend 조합은 중복될 수 없다.

## API 인증 구분
- `POST /friends`는 AccessToken 인증이 필요한 보호 API다.
- `GET /friends`는 AccessToken 인증이 필요한 보호 API다.

## 영속성 전략
- 친구 관계 저장과 중복 여부 검증은 JPA Repository를 사용한다.
- 친구 목록 조회는 MyBatis Mapper를 사용한다.
- 친구 목록 조회는 응답에 필요한 friendId, friendUsername만 조회한다.
- 친구 목록 조회 시 삭제되지 않은 친구 유저만 조회한다.
- 친구 목록은 친구 유저명을 기준으로 오름차순 정렬한다.
- 채팅방 생성과 초대에서는 친구 관계 검증을 위해 MyBatis Mapper의 active friend count 조회를 사용한다.

## 요청 흐름

### 1. 친구 추가
- 대상 엔드포인트: `POST /friends`

동작:
1. AccessToken 인증 정보에서 로그인 유저 ID를 추출한다.
2. 요청값을 검증한다.
3. 로그인 유저가 존재하는지 검증한다.
4. 로그인 유저가 삭제되지 않았는지 검증한다.
5. 요청한 username으로 친구 추가 대상 유저를 조회한다.
6. 친구 추가 대상 유저가 존재하는지 검증한다.
7. 친구 추가 대상 유저가 삭제되지 않았는지 검증한다.
8. 자기 자신을 친구로 추가하는 요청인지 검증한다.
9. 이미 친구로 등록된 유저인지 검증한다.
10. 친구 관계를 저장한다.

결과:
1. 로그인 유저의 친구 목록에 대상 유저가 추가된다.
2. 대상 유저의 친구 목록에는 로그인 유저가 자동으로 추가되지 않는다.
3. 응답 바디 없이 성공 응답을 반환한다.

### 2. 친구 목록 조회
- 대상 엔드포인트: `GET /friends`

동작:
1. AccessToken 인증 정보에서 로그인 유저 ID를 추출한다.
2. 로그인 유저가 존재하는지 검증한다.
3. 로그인 유저가 삭제되지 않았는지 검증한다.
4. 로그인 유저 ID를 기준으로 친구 관계를 조회한다.
5. 삭제되지 않은 친구 유저만 조회한다.
6. 친구 목록을 유저명 기준으로 정렬한다.
7. 친구 ID와 친구 유저명을 응답한다.

결과:
1. 로그인 유저가 추가한 친구 목록을 조회할 수 있다.
2. 삭제된 친구 유저는 목록에 포함되지 않는다.
3. 친구 관계가 없는 경우 빈 목록을 응답한다.

### 3. 친구 관계 활용
- 사용 위치: 채팅방 생성, 채팅방 초대

동작:
1. 요청한 대상 유저 ID 목록을 기준으로 친구 관계를 검증한다.
2. 로그인 유저가 owner인 친구 관계만 인정한다.
3. 삭제되지 않은 친구 유저만 유효한 친구로 인정한다.
4. 필요한 친구 수와 실제 조회된 active friend 수가 일치하는지 확인한다.

결과:
1. 친구가 아닌 유저는 채팅방 생성 또는 초대 대상이 될 수 없다.
2. 삭제된 유저는 친구 관계가 남아 있어도 유효한 초대 대상이 될 수 없다.

## 기대 효과
- 친구 관계를 단방향으로 관리하여 유저별 친구 목록을 독립적으로 유지할 수 있다.
- 친구 추가 시 자기 자신 추가와 중복 추가를 차단할 수 있다.
- 삭제된 유저를 친구 추가와 친구 목록 조회에서 제외할 수 있다.
- 채팅방 생성과 초대에서 친구 관계를 재사용하여 접근 가능한 유저 범위를 제한할 수 있다.
- 친구 목록 조회는 필요한 컬럼만 조회하여 응답 DTO 구성에 맞게 처리할 수 있다.

## 주의사항
- 친구 관계는 단방향 관계다.
- 현재 친구 삭제 API는 제공하지 않는다.
- 이미 저장된 친구 관계의 friend 유저가 삭제되면 관계 row는 남지만 친구 목록 조회에서는 제외된다.
- owner와 friend 조합에는 DB unique 제약이 있으므로 같은 친구 관계를 중복 저장할 수 없다.
- 서비스 계층에서 중복 검사를 수행하지만, 동시 요청에 대한 최종 중복 방어는 DB unique 제약이 담당한다.
- 친구 관계 검증은 로그인 유저가 owner인 관계만 기준으로 한다.
- 타 도메인에서 친구 관계를 사용할 때도 삭제된 유저를 유효한 친구로 취급하면 안 된다.
