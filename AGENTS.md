# AGENTS.md

## 목적
이 파일은 이 저장소에서 작업하는 사람과 AI 에이전트를 위한 진입점이다.
이 파일은 길게 설명하는 문서가 아니라, 작업 기준과 탐색 순서를 빠르게 알려주는 안내서다.

## 프로젝트 개요
- Java 17 / Spring Boot 3.5 기반 채팅 백엔드다.
- 핵심 도메인은 `user`, `auth`, `friend`, `chat` 이다
- 실시간 통신은 WebSocket(STOMP) + Redis Pub/Sub를 사용한다.
- 영속성 계층은 MySQL, Spring Data JPA, MyBatis를 함께 사용한다.

## 빌드와 실행 규칙
- 빌드와 테스트는 반드시 Gradle Wrapper(`./gradlew`, `gradlew.bat`)로 실행한다.
- 기본 실행 및 검증 환경은 `SPRING_PROFILES_ACTIVE=local` 기준으로 본다.
- 로컬 작업 시 MySQL과 Redis는 `application-local.yml` 설정값에 맞게 이미 실행 중이라고 가정한다.
- 사용자의 명시적인 요청이 없으면 에이전트가 임의로 Docker Compose를 실행하거나 중지하지 않는다.
- `SECRET`는 최소 32바이트 이상의 문자열로 설정한다.

## 저장소 구조
- `build.gradle`: 의존성과 빌드 진입점
- `src/main/java/com/tgg/chat/common`: security, websocket, redis, swagger, 공통 설정 등 공유 인프라
- `src/main/java/com/tgg/chat/domain/<domain>`: 도메인별 비즈니스 로직
- `src/main/java/com/tgg/chat/exception`: 공통 예외 코드, 커스텀 예외, 에러 응답, 전역 예외 처리
- `src/main/resources/mappers/**`: MyBatis XML 매퍼
- `src/main/resources/application*.yml`: 프로필별 실행 설정
- `src/test/java/**`: 자동화 테스트
- `docker-compose.yml`: 로컬 인프라 및 컨테이너 실행 설정
- `.github/workflows/deploy.yml`: 컨테이너 빌드 및 배포 워크플로우

## 작업 원칙
- 사용자의 명시적인 수정 요청이 없으면 우선 코드 변경 없이 분석, 원인 파악, 개선안을 먼저 제안한다.
- 사용자가 `변경사항 확인`, `변경사항 리뷰`, `커밋메시지 추천`을 요청하면 특별한 언급이 없는 한 기본적으로 `staged changes`를 기준으로 판단한다.
- `unstaged` 변경이 남아 있어도 사용자가 별도로 요청하지 않으면 검토 대상과 커밋 메시지 추천 범위에 포함하지 않는다.
- 코드를 수정하기 전에 관련 `controller`, `service`, `repository`, `entity`, `mapper`를 먼저 읽는다.
- 넓은 재작성보다 안전한 최소 변경을 우선한다.
- 코드, 설정, 프레임워크 사용 방식을 사용자에게 추천하기 전에는 공식 문서, 검증된 외부 사례, 프로젝트 이슈 등 실제 근거를 먼저 확인하고, 근거 없이 추측성 제안을 하지 않는다.
- 코드나 구조 변경안을 추천할 때는 해당 방식을 적용했을 때 생길 수 있는 오류, 자동 등록, 중복 실행, 예외 처리 누락, 호환성 문제 같은 부작용과 사이드 이펙트를 사전에 조사하고 함께 설명한다.
- 비즈니스 로직은 해당 도메인 패키지 안에 둔다.
- 비즈니스 규칙을 `common/**`에 넣지 않는다.
- 명시적 리팩터링 요청이 없는 한 현재 아키텍처를 유지한다.
- 새로운 프레임워크, 라이브러리, 공통 패턴은 근거 없이 추가하지 않는다.
- 비밀값, 비밀번호, 토큰, 로컬 머신 의존 설정은 커밋하지 않는다.

## 아키텍처 제약
- `controller`: 요청 매핑, 입력 검증, 인증 주체 추출, 응답 구성만 담당한다. 얇게 유지한다.
- `service`: 비즈니스 규칙, 유스케이스 조합, 트랜잭션 경계를 담당한다.
- `repository` / `mapper`: 영속성 처리만 담당한다.
- `entity`: 영속 모델과 도메인 불변식을 표현하며, 전송 계층 관심사를 넣지 않는다.
- 예외 처리는 `com.tgg.chat.exception` 패키지를 기준으로 일관되게 관리한다. 
- 기본적으로 JPA를 우선적으로 사용하되 복잡한 조회 쿼리는 MyBatis를 사용하도록 한다. 
- 같은 영속 책임을 JPA와 MyBatis 양쪽에 중복 구현하지 않는다. 필요한 경우 이유를 남긴다.
- JWT, Spring Security filter, WebSocket 인증, Redis publish/subscribe, 채팅 메시지 순서 보장은 고위험 변경으로 취급하고 명시적으로 검증한다.

## 검증 원칙
- 테스트 코드는 기본적으로 사용자의 요청 범위를 따른다. 단, 버그 수정이나 핵심 비즈니스 로직 변경 시에는 회귀 방지를 위한 테스트 추가를 우선 제안한다.
- 사용자가 명시적으로 코드 수정을 요청한 경우에는 build를 통해 전체 무결성을 확인한다.

## Harness Engineering (시스템 개선 원칙)
- 동일한 오류나 실패가 2회 이상 반복되면, 코드를 무작정 다시 고치지 말고 테스트, 스크립트 등 검증 환경(Harness)의 부재를 의심하고 사용자에게 환경 개선을 먼저 제안한다.
- 반복적인 지시나 잦은 실수가 발생하면, 이를 프롬프트에 구두로 남기지 말고 `문서`, `스크립트`, `테스트`, `검증 규칙(Lint 등)` 같은 영구적인 기계적 산출물로 고정한다.
- 복잡한 기능 개발이나 리팩터링 시, 코드를 작성하기 전에 반드시 `docs/exec-plans/`에 목표, 제약 사항, 변경 예정 파일, 검증 방법을 짧게 기록하고 사용자의 승인을 받는다.
- 상세한 배경지식이나 도메인 설명은 `docs/` 디렉토리로 분리하여 참조한다.

## 컨텍스트 및 문서 활용 (`docs/`)
- 코드를 작성하거나 수정하기 전에 반드시 `docs/` 디렉토리를 탐색하여 해당 도메인, 아키텍처, 인프라와 관련된 기존 문서가 있는지 확인한다.
- 관련 문서가 존재하면 우선 그 흐름과 제약사항을 참고하되, 코드와 문서가 불일치하면 코드를 기준으로 판단하고 문서의 차이도 함께 정리한다.
- 로그인, 로그아웃, 토큰 재발급, `/login-status`, Redis 토큰 저장/검증처럼 인증 흐름과 직접 관련된 작업을 할 때는 먼저 `docs/AUTHFLOW.md`를 읽고 단일 세션 정책과 토큰 관리 전략의 맥락부터 파악한다.
- 기능 수정 및 새 기능 작성 시, 관련 docs가 이미 존재한다면 코드와 함께 문서를 수정하고, 없다면 새로 생성할 것을 제안한다.
- 빌드, 테스트, 실행 중 실패가 발생하면 먼저 `docs/FAIL.md`를 확인하여 기존에 기록된 원인인지부터 검증한다.
- `docs/FAIL.md`에 있는 기존 원인이면 기록된 해결 절차를 먼저 적용하고, 새로운 원인이면 날짜, 재현 조건, 원인, 해결 방안을 같은 문서에 추가한다.

## 작업 완료 체크리스트
작업 종료 전 아래를 확인한다.
1. 관련 패키지와 흐름을 읽었다.
2. 가장 작은 안전한 변경을 적용했다.
3. 적절한 범위의 검증을 실행했다.
4. 동작이나 제약이 바뀌면 테스트와 문서를 갱신했다.
5. 비밀값이나 환경 종속값을 새로 추가하지 않았다.
